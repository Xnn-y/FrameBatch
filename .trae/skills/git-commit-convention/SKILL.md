# Git 提交规范

当需要为用户准备提交、创建提交、检查暂存区或拆分提交时，使用这个 skill。

## 核心规则

- 每个提交只包含一个明确主题，方便回滚和追踪。
- 不要把无关改动混进同一个提交。
- 使用 Conventional Commits 格式：`<type>(<scope>): <subject>`。
- `subject` 要简洁、具体，末尾不要加句号。
- 当用户说"提交推送"或要求"提交并推送"时，提交标题的 `subject` 默认使用中文，除非用户明确指定其他语言。
- 用户没有明确要求时，不要执行 `git push`。
- 不要还原、丢弃用户改动，除非用户明确要求。

## 必须遵循的流程

按顺序执行：

1. 运行 `git status` 检查工作区状态。
2. 运行 `git diff` 查看未暂存改动。
3. 识别提交主题；如果改动混杂，先拆分提交。
4. 只暂存当前主题相关文件，使用 `git add <file>` 或 `git add -p`。
5. 运行 `git diff --staged` 检查即将提交的内容。
6. 使用 `git commit -m "<type>(<scope>): <subject>"` 创建提交。
7. 再次运行 `git status`，并向用户报告结果。

## 提交信息格式

使用：

```text
<type>(<scope>): <subject>
```

允许的 `type`：

- `feat`：新增用户可见功能
- `fix`：修复缺陷
- `refactor`：内部结构调整，不改变行为
- `perf`：性能优化
- `style`：纯格式调整
- `docs`：仅文档改动
- `test`：测试相关改动
- `build`：构建系统或依赖改动
- `ci`：持续集成配置改动
- `chore`：维护性工作
- `revert`：回滚提交

`scope` 从受影响的模块、包、应用、目录或文件族中选择。

示例：

```text
feat(auth): 新增登录令牌自动刷新
fix(order): 修复空状态筛选导致的查询异常
docs(readme): 补充本地启动步骤说明
chore(graphify): 添加中文图谱输出文件
fix(api): 修复空响应处理异常
```

## 拆分提交指引

如果工作区包含多个主题，创建多个提交。通常需要拆开的主题包括：

- 源码行为改动
- 测试改动
- 文档改动
- 生成产物
- 纯格式化改动
- 依赖或构建配置改动

如果存在生成产物，需要明确说明，并且只在它属于本次请求时提交。

## 安全检查

提交前确认：

- 暂存区 diff 只对应一个清晰主题
- 没有暂存密钥、本地私有文件或临时文件
- 没有暂存无关的用户改动
- 提交信息使用允许的 `type`，并且 `subject` 清晰

如果仓库里有明显相关且运行较快的测试或检查，提交前运行它们。如果跳过检查，需要在最终回复里说明原因。

## 最终回复

提交完成后，向用户报告：

- 提交 hash 和提交信息
- 本次提交包含的文件或主题
- 测试/检查结果，或说明为什么没有运行
- 是否还有未提交改动
